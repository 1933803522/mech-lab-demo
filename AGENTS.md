# AGENTS.md

本文件用于约束 AI 助手在本仓库中的协作方式。所有 AI 助手在修改、创建或整理本仓库内容前，都应优先遵守本文件。

## 适用范围

本文件用于记录本仓库中逐步沉淀的 AI 协作规则。每条规则的具体适用范围以规则正文为准。

## 规则 1：写 PRD 时不要编造产品事实

### 1.1 适用范围

本规则仅适用于 PRD 写作、PRD 修改、PRD 整理和 PRD 评审相关工作。

### 1.2 基本原则

- 写 PRD 时不要编造产品事实。
- 不要把项目管理目标、AI 协作流程、README 中的阶段规划，直接当作产品 PRD 的事实内容。
- 没有被用户明确提供或确认的业务信息，不应写入正式 PRD。
- 可以整理结构、提炼已确认信息、列出待确认问题，但不能替用户决定产品范围、页面结构、用户角色、业务流程、数据模型或接口方案。

### 1.3 PRD 写作规则

在编写或修改 PRD 时：

- 只写用户已明确提供的信息。
- 如果内容缺失，使用“待确认”标记，或先向用户提问。
- 不要根据项目名称自行推断页面、模块、角色、流程、字段、状态、权限或后端接口。
- 不要为了让文档看起来完整而补充未经确认的内容。
- 可以创建空结构或模板，但模板中的业务内容必须保持空白或标注“待补充”。
- 如果需要基于推断提出建议，必须明确写成“建议”或“可选方案”，并等待用户确认后再写入正式文档。

### 1.4 PRD 协作规则

- PRD 是产品本身的需求文档，不等同于本仓库的 AI 实验计划。
- AI 工作流、提示词、复盘内容应放入 `AI协作总结/`，不要混入产品 PRD，除非用户明确要求。
- 产品 PRD 应聚焦“力学实验低代码平台”本身，产品事实必须来自用户复盘或用户确认。
- 参考 `PRD写作基本要求清单.md` 时，只参考文档结构和质量要求，不要据此生成未经确认的业务内容。
- 在用户没有提供具体业务事实时，优先输出问题清单，而不是直接生成完整 PRD。

### 1.5 写入 PRD 前的检查

写入 PRD 前，先确认：

- 这些内容是否来自用户明确描述？
- 这些内容是否只是 AI 推断？
- 如果是推断，是否已经标注为建议并得到用户确认？
- 是否把 AI 协作流程误写进了产品 PRD？
- 是否存在为了补全文档而虚构页面、流程、接口或数据的情况？

如果以上任一问题无法确认，应先停下来向用户提问。
